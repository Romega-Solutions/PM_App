# PinayMate Supabase Messaging Integration

Status: launch-stage implementation runbook. The app code uses Supabase-backed messaging paths, but production completion still requires live database, storage, realtime, and device verification.

## Current architecture

```text
MessagesScreen
  -> useConversations
  -> conversations.api.ts
  -> get_user_conversations RPC

ChatScreen
  -> useMessages
  -> messages.api.ts
  -> send_message RPC
  -> messages table

Image message
  -> useMessageUpload
  -> uploadChatImage
  -> chat-images storage bucket
  -> requires an existing conversation id
  -> messages.image_url stores durable storage path
  -> hydrateMessageMedia signs URL for display

Realtime
  -> useChatRealtime
  -> realtime.api.ts
  -> Supabase channel events
```

## Implemented app paths

- `sendTextMessage` calls `send_message`; the backend creates or reuses the conversation when the first valid text is sent.
- `sendImageMessage` calls `send_message` with an existing conversation-bound storage path after upload.
- `getMessages` and `getMessagesByConversationId` hydrate image media before display.
- `markMessagesAsRead` uses `mark_messages_read` RPC.
- `markConversationAsRead` uses `mark_conversation_read` RPC.
- `updateMessageStatus` uses `update_message_status` RPC.
- `deleteMessage` uses `delete_message_for_me` or `delete_message_for_everyone` RPC.
- `getUnreadCount` reads unread message count for the current user.
- `uploadChatImage` uploads to `chat-images` and returns a durable storage path.
- `deleteChatImage` removes storage objects by durable path or signed/public URL form.
- `getConversationsForUser` uses `get_user_conversations` RPC.

## Security model

Messaging must be protected by both app logic and Supabase database policy.

Required controls:

- only matched users can create/open conversations
- app clients cannot call `get_or_create_conversation` directly
- empty visible conversations must not be created without a real message
- blocked pairs cannot message each other
- users can read only conversations they participate in
- clients cannot directly forge message status, delete state, or conversation read state
- image storage writes are scoped to authenticated users and valid conversations
- image display uses signed URLs instead of public object URLs
- report/block/unmatch flows go through safety RPCs

## Storage behavior

Chat image messages should store a durable path such as:

```text
<conversation-id>/<timestamp>_<user-id>.jpg
```

The UI should render a signed URL generated by `createSignedUrl`. Public bucket URLs are not launch-proof for private chat media.

## Intentionally unavailable features

Voice and video call buttons do not start calls. They open screens that explain a real calling provider is required. This is intentional and safer than simulating calls.

The following features are roadmap-only unless separately implemented and verified:

- voice calls
- video calls
- push notifications
- offline queue
- reactions
- forwarding
- general file attachments
- moderation dashboard

## Required Supabase artifacts

The target environment must include:

- `messages` table
- `conversations` table
- `chat-images` storage bucket
- `get_or_create_conversation` RPC as a private helper for `send_message`; direct client execution must be denied
- `get_user_conversations` RPC
- `mark_messages_read` RPC
- `mark_conversation_read` RPC
- `update_message_status` RPC
- `delete_message_for_me` RPC
- `delete_message_for_everyone` RPC
- `block_user` RPC
- `unmatch_user` RPC
- `submit_user_report` RPC
- RLS policies that deny unmatched/blocked access
- storage policies that deny blocked/unmatched chat image access

## Verification gates

Local code checks:

```powershell
npm test -- --runInBand
npx tsc --noEmit
npm run lint
npm run build:web
```

Database/security checks:

```powershell
psql $env:DATABASE_URL -v ON_ERROR_STOP=1 -f .\supabase\tests\04_safety_smoke_test.sql
```

The smoke/preflight proof must show that authenticated clients cannot execute `get_or_create_conversation` directly while `send_message` still creates conversations for valid matched text sends.

Native QA checks are in `docs\testing\NATIVE_QA_SCRIPT.md`.

## Production blockers

Do not mark messaging production-ready until:

- migrations are applied in staging and production
- smoke SQL passes against the target database
- chat image upload and signed display pass on device/emulator
- blocked/unmatched messaging denial is proven live
- support and moderation owners are assigned
- PM_Web does not claim live messaging beyond actual availability

## Current evidence limitation

Static code checks can prove the app is wired to the backend API surface. They cannot prove Supabase RLS, storage, realtime, or native permission behavior in production. Those require live environment evidence.
